# Fallback Flow

## Основной поток

1. Пользователь открывает блок рекомендаций.
2. Frontend отправляет запрос в Recommendation Gateway.
3. Gateway собирает контекст пользователя и поездки.
4. Gateway отправляет запрос в ML-сервис с timeout 3 секунды.
5. Если ML отвечает быстро и уверенно, возвращается ML-выдача.
6. Если ML недоступен, отвечает слишком долго или возвращает low confidence, включается fallback engine.
7. Fallback engine строит подборку на внутренних данных.
8. Frontend получает единый API-ответ.
9. Пользователь видит рекомендации без технической ошибки.

## Почему нужен Gateway

Если frontend напрямую зависит от ML-сервиса, отказ модели ломает интерфейс. Recommendation Gateway решает эту проблему: он скрывает источник рекомендаций и возвращает один контракт для всех режимов.

## Circuit breaker

Если ML-сервис часто падает или отвечает нестабильно, circuit breaker временно прекращает обращения к нему. Это снижает нагрузку на систему и ускоряет включение fallback.

## Hybrid-режим

Hybrid нужен, когда ML частично полезен. Например, первые 3 рекомендации имеют высокий confidence, а остальные слабые. В таком случае сильные позиции ML можно оставить, а слабые заменить fallback-кандидатами.
