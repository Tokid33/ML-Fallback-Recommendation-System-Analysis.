# Non-Functional Requirements

## Производительность

- Пользователь не должен ждать ML дольше 3 секунд.
- После timeout система должна вернуть fallback-рекомендации.
- Fallback engine должен отвечать достаточно быстро, чтобы не создавать вторую задержку после ML timeout.

## Надёжность

- Отказ ML-сервиса не должен ломать блок рекомендаций.
- Recommendation Gateway должен поддерживать fallback-режим.
- При серии ошибок ML должен включаться circuit breaker.

## Приватность

- Fallback использует только внутренние данные и агрегированную анонимную статистику.
- Нельзя раскрывать поведение конкретных пользователей.
- Объяснения рекомендаций должны быть нейтральными и безопасными.

## Совместимость

- ML, fallback и hybrid возвращают один API-контракт.
- Frontend не должен реализовывать отдельную логику для каждого режима.
- Поле `mode` используется для диагностики и аналитики, но не должно ломать UI.

## Наблюдаемость

- Нужно логировать режим ответа.
- Нужно логировать причину fallback.
- Нужно измерять время ответа ML и fallback.
- Нужно связывать технические метрики с продуктовыми: CTR, booking conversion, post-trip feedback.
