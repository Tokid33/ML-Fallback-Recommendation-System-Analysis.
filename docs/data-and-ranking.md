# Data and Ranking

## Источники данных для fallback

Fallback не использует заранее сохранённые ML-предсказания. Он строит выдачу на момент запроса.

Используемые данные:

- профиль пользователя;
- история поездок;
- просмотры;
- сохранения;
- бронирования;
- оценки после поездки;
- текущий маршрут;
- даты и длительность поездки;
- выбранный отель;
- свободные окна в расписании;
- состав поездки;
- агрегированная анонимная популярность.

## Пример score

Итоговый score можно считать как взвешенную сумму факторов:

```text
score =
  interest_match * 0.25 +
  budget_match * 0.15 +
  distance_match * 0.15 +
  schedule_fit * 0.20 +
  anonymous_popularity * 0.15 +
  rating_score * 0.10
```

Веса можно менять по результатам продуктовых метрик и A/B-тестов.

## Ограничения ранжирования

Нельзя показывать активность, если она конфликтует с уже забронированным событием.

Нельзя показывать вариант, который недоступен на даты поездки.

Нужно избегать выдачи из 10 одинаковых вариантов. Для этого используется diversity rule: в подборке должны быть разные типы рекомендаций.

## Пример логики

Пользователь едет в Рим с семьёй на 5 дней. В истории у него музеи, прогулки, спокойные маршруты и отели среднего/высокого сегмента.

Fallback должен повысить:

- музеи;
- обзорные прогулки;
- семейные активности;
- рестораны рядом с отелем;
- маршруты без конфликтов по времени.

Fallback должен понизить:

- ночные клубы;
- экстремальные активности;
- слишком дорогие варианты;
- активности далеко от маршрута;
- события, которые конфликтуют с расписанием.
