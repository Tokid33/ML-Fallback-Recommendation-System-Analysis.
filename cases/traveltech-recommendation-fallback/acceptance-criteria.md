# Acceptance Criteria

## Scenario 1: ML service is unavailable

Given the ML recommendation service returns HTTP 5xx or is unreachable

When the user opens the recommendation block

Then Recommendation Gateway returns fallback recommendations

And frontend displays recommendations without a technical error

And logs contain `reason = ml_unavailable`

And response `mode = fallback`

## Scenario 2: ML response takes more than 3 seconds

Given the ML recommendation service does not return a response within 3 seconds

When Recommendation Gateway waits for ML output

Then Gateway stops waiting for ML for the user-facing response

And returns fallback recommendations

And logs contain `reason = ml_timeout`

And frontend receives the same response contract as in ML mode

## Scenario 3: ML confidence is below threshold

Given the ML service returns recommendations with confidence below 0.6

When Recommendation Gateway validates the ML response

Then the system uses fallback or hybrid mode

And weak ML recommendations are not shown without replacement

And logs contain `reason = low_confidence`

## Scenario 4: ML returns after fallback is already displayed

Given fallback recommendations are already visible to the user

When ML returns a high-quality response later

Then the visible list is not abruptly reordered

And the user may see a soft update option

And the user can continue the current flow without losing context

## Scenario 5: Group trip recommendations

Given the trip has multiple participants

When fallback recommendations are generated

Then the engine considers interests and constraints of multiple participants

And options with schedule conflicts are downgraded or removed

And compromise activities may be ranked higher

## Scenario 6: Unified frontend contract

Given recommendations are generated by ML, fallback or hybrid mode

When frontend receives the response

Then the response structure is the same for all modes

And frontend does not need separate rendering logic for each mode

## Scenario 7: Privacy-safe popularity

Given fallback uses popularity signals

When the system explains a recommendation

Then it uses only aggregated anonymous wording

And it does not expose actions of specific users
