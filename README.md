# TravelTech Recommendation Fallback Case

## Резервный сценарий рекомендаций при отказе ML-сервиса

## 1. Контекст

TravelTech-сервис помогает пользователю планировать путешествие: подобрать билеты, отели, трансферы, активности и собрать поездку в едином интерфейсе. В продукте есть ML-модель, которая персонализирует рекомендации по отелям, активностям и маршрутам.

Рекомендации важны для продукта, потому что они влияют на глубину планирования поездки, вовлечённость пользователя и конверсию в бронирование.

Проблема: ML-сервис может быть недоступен, отвечать слишком долго или возвращать рекомендации с низкой уверенностью. Если весь UX завязан только на ML, пользователь увидит пустой блок, ошибку или слишком долгую загрузку.

Цель кейса: спроектировать резервный сценарий, который сохраняет пользовательский опыт даже при проблемах с ML-сервисом.

## 2. Моя роль

В рамках кейса я выступал в роли системного / бизнес-аналитика:

- разобрал продуктовый сценарий персональных рекомендаций;
- выделил отказные ситуации ML-сервиса;
- описал fallback-логику без кэширования ML-предсказаний заранее;
- спроектировал Recommendation Gateway как слой между frontend и источниками рекомендаций;
- описал режимы ML / fallback / hybrid;
- подготовил API-ответ, критерии приёмки, метрики, NFR и UX-правила;
- учёл ограничения по приватности и использованию внутренних данных.

## 3. Бизнес-проблема

Персональные рекомендации помогают пользователю быстрее собрать поездку и выбрать подходящие варианты. Если блок рекомендаций ломается, продукт теряет часть ценности в момент принятия решения.

Для бизнеса отказ рекомендаций может привести к:

- снижению CTR по карточкам рекомендаций;
- падению конверсии в бронирование;
- росту отказов на этапе планирования;
- ухудшению доверия к продукту;
- зависимости важного UX-сценария от одного ML-сервиса.

Задача не в том, чтобы “починить ML”, а в том, чтобы продукт продолжал работать даже при деградации ML-компонента.

## 4. Когда включается fallback

Fallback-режим включается в трёх случаях:

1. ML-сервис недоступен:
   - ошибка 5xx;
   - network error;
   - circuit breaker открыт;
   - сервис не отвечает.

2. ML-сервис отвечает дольше 3 секунд:
   - backend не ждёт модель бесконечно;
   - пользователю возвращается резервная подборка.

3. ML вернул результат с низким confidence:
   - confidence ниже заданного порога, например 0.6;
   - результат считается недостаточно надёжным для показа без дополнительной обработки.

Во всех случаях пользователь не должен видеть техническую ошибку.

## 5. Ограничения

Главное ограничение: нельзя заранее кэшировать предсказания ML-модели, потому что интересы пользователя быстро устаревают.

Fallback должен строить рекомендации заново на момент запроса и использовать только внутренние данные:

- профиль пользователя;
- историю поездок;
- просмотры, сохранения и бронирования;
- текущий контекст поездки;
- уже выбранные билеты, отели и активности;
- анонимную агрегированную популярность среди пользователей;
- сезонность, направление и тип поездки.

Нельзя раскрывать поведение конкретных пользователей. Допустимо использовать агрегированные формулировки вроде “популярно у путешественников в этом городе”, но нельзя показывать персональные данные других людей.

## 6. Предложенное решение

Я предлагаю выделить отдельный слой: Recommendation Gateway.

Frontend не должен знать, откуда пришли рекомендации: из ML, fallback или hybrid-режима. Он всегда получает единый формат ответа.

Recommendation Gateway отвечает за:

- приём запроса рекомендаций от frontend;
- сбор контекста пользователя и поездки;
- отправку запроса в ML-сервис с timeout 3 секунды;
- проверку доступности ML-сервиса;
- проверку confidence результата;
- включение fallback engine при необходимости;
- возврат рекомендаций в едином формате;
- логирование режима и причины fallback.

Fallback engine работает как rule-based контур: он не пытается заменить ML полностью, но даёт достаточно релевантную подборку на основе внутренних данных и понятных правил ранжирования.

## 7. Какие данные использует fallback

### Данные пользователя

- интересы из профиля;
- предпочитаемый бюджет;
- тип отдыха;
- история поездок;
- сохранённые места;
- просмотры и бронирования;
- оценки после поездок.

### Контекст текущей поездки

- город или страна поездки;
- даты;
- длительность поездки;
- состав поездки: один человек, пара, семья, группа;
- выбранный отель;
- выбранные билеты;
- свободные временные окна;
- уже добавленные активности;
- тайм-зона и локальные форматы времени.

### Агрегированные данные

- популярные активности по направлению;
- популярные отели по сезону;
- рейтинг и количество отзывов;
- частота бронирований;
- анонимная популярность среди пользователей с похожим контекстом поездки.

## 8. Логика ранжирования fallback-рекомендаций

Для каждого кандидата fallback engine считает score.

В score могут входить:

- совпадение с интересами пользователя;
- соответствие бюджету;
- близость к отелю или текущему маршруту;
- отсутствие конфликта по времени;
- доступность на выбранные даты;
- рейтинг и количество отзывов;
- анонимная популярность;
- соответствие составу поездки;
- разнообразие выдачи.

Пример:

Если пользователь раньше выбирал музеи, спокойные прогулки и отели 4 звезды, а текущая поездка семейная, fallback должен понижать ночные клубы и экстремальные активности. Вверху выдачи должны оказаться музеи, обзорные маршруты, семейные активности, рестораны рядом с отелем и варианты без конфликта с расписанием.

Для групповой поездки можно использовать объединённый профиль: учитывать интересы участников, искать пересечение интересов и понижать конфликтные варианты.

## 9. Режимы рекомендаций

### ML mode

Используется, если ML-сервис ответил быстрее 3 секунд и вернул confidence выше порога.

### Fallback mode

Используется, если ML-сервис недоступен, ответил слишком долго или вернул низкий confidence.

### Hybrid mode

Используется, если часть ML-рекомендаций достаточно уверенная, а часть слабая. Сильные позиции ML остаются, слабые позиции заменяются fallback-кандидатами.

## 10. Плавное переключение UX

Пользователь не должен видеть технические детали отказа ML.

Правила UX:

- во время ожидания показывается skeleton/loading;
- если ML не ответил за 3 секунды, показывается fallback-подборка;
- если ML позже вернул качественный ответ, текущий список не меняется резко;
- можно показать мягкое обновление: “Мы нашли ещё более подходящие варианты” и кнопку “Обновить подборку”;
- если confidence низкий, пользователь видит fallback или hybrid без сообщения об ошибке.

Нейтральная формулировка для интерфейса:

> Подобрали варианты на основе вашего профиля и популярности у путешественников.

## 12. Логирование и метрики

Нужно логировать:

- режим ответа: ml / fallback / hybrid;
- причину fallback: ml_unavailable / ml_timeout / low_confidence;
- время ответа ML-сервиса;
- время ответа fallback engine;
- confidence ML-модели;
- количество рекомендаций;
- CTR по рекомендациям;
- конверсию в бронирование;
- скрытия и жалобы на рекомендации;
- оценку рекомендаций после поездки.

Ключевые метрики:

- fallback rate;
- среднее время ответа Recommendation Gateway;
- timeout rate;
- low confidence rate;
- recommendation CTR;
- booking conversion;
- post-trip feedback score.

## 13. Нефункциональные требования

### Производительность

Пользователь не должен ждать ML дольше 3 секунд. После превышения порога включается fallback.

### Надёжность

Отказ ML-сервиса не должен ломать пользовательский сценарий. Gateway должен поддерживать circuit breaker.

### Приватность

Fallback использует только внутренние данные и агрегированную анонимную статистику. Поведение конкретных пользователей не раскрывается.

### Совместимость

ML, fallback и hybrid должны возвращать единый формат ответа для frontend.

### Наблюдаемость

Система должна логировать режим, причину fallback, время ответа и метрики качества.

## 14. Критерии приёмки

Подробные критерии описаны в файле [`acceptance-criteria.md`](acceptance-criteria.md).

Ключевые сценарии:

- ML недоступен -> показывается fallback;
- ML отвечает дольше 3 секунд -> показывается fallback;
- ML вернул low confidence -> используется fallback или hybrid;
- ML ответил после показа fallback -> список не перестраивается резко;
- групповая поездка -> учитываются интересы нескольких участников.

## 15. Ожидаемый бизнес-эффект

Решение снижает зависимость ключевого продуктового сценария от ML-сервиса.

Для бизнеса это даёт:

- непрерывность пользовательского опыта;
- снижение отказов в блоке рекомендаций;
- защиту конверсии в бронирование;
- возможность отдельно измерять качество ML и fallback;
- более безопасное использование персонализации;
- управляемую деградацию вместо полного отказа функции.
